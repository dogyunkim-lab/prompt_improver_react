# Prompt Improver 사용자 가이드

> **LLM 요약 프롬프트를 반복 실험으로 개선하는 도구**
>
> 콜센터 상담 STT(음성인식 텍스트)를 요약하는 LLM 프롬프트의 품질을 측정하고, 자동으로 개선 후보를 생성하며, Dify 워크플로우에서 실행·재판정하여 점수를 비교합니다.
> 이 과정을 반복하여 **LLM Judge 기준 정답+과답 ≥ 95%** 를 달성하는 것이 목표입니다.

## 1. 시스템 개요

### 이 도구가 해결하는 문제

콜센터 상담 녹음을 STT로 변환한 뒤 LLM이 요약합니다. 그런데 LLM 요약 품질이 충분하지 않습니다. "어떤 케이스에서 왜 틀렸는지" 분석하고, "프롬프트를 어떻게 고쳐야 하는지" 자동으로 제안하며, "고친 프롬프트로 다시 실행하면 실제로 좋아졌는지" 측정하는 반복 실험 사이클이 필요합니다.

이 도구는 그 전체 사이클을 **6단계 자동화 파이프라인**으로 제공합니다.

### 기술 스택

| 구성요소 | 기술 |
|---------|------|
| 백엔드 | FastAPI + SQLite (aiosqlite) |
| 프론트엔드 | React 18 + TypeScript + Zustand + Tailwind CSS v3 |
| 분석/Judge 모델 | gpt-oss-120B (OpenAI 호환 API, 폐쇄망) |
| 생성 모델 (워크플로우) | Dify API (Qwen3-30B 등) |
| 시뮬레이션 모델 (Mini-validation) | Task 설정에서 지정 (Qwen 등) |

### 아키텍처 다이어그램

```
┌─────────────────────────────────────────────────────────────────┐
│  사용자 (브라우저)                                                │
│  React SPA ← SSE(Server-Sent Events) 실시간 스트리밍             │
└──────────────────────┬──────────────────────────────────────────┘
                       │ REST API + SSE
┌──────────────────────▼──────────────────────────────────────────┐
│  FastAPI 백엔드 (main.py)                                        │
│  ├── routers/tasks.py    ← Task CRUD                            │
│  ├── routers/runs.py     ← Run 생성/조회, 파일 업로드             │
│  ├── routers/phases.py   ← Phase 1~6 실행/스트리밍               │
│  ├── services/phase1_analysis.py                                │
│  ├── services/phase2_design.py                                  │
│  ├── services/phase3_dify.py                                    │
│  ├── services/phase4_judge.py                                   │
│  ├── services/phase6_strategy.py                                │
│  └── services/delta.py                                          │
└──────────┬──────────────────────┬───────────────────────────────┘
           │                      │
    ┌──────▼──────┐        ┌──────▼──────┐
    │ gpt-oss-120B │        │  Dify API   │
    │ (분석/Judge) │        │ (워크플로우) │
    └─────────────┘        └─────────────┘
```

## 2. 설치 및 실행

### 사전 요구사항

- Python 3.10+
- Node.js 18+ (Tailwind v3 호환)
- gpt-oss-120B API 접근 가능한 네트워크
- Dify 워크플로우 플랫폼 접근

### 백엔드 실행

```bash
cd prompt-improver-react
pip install -r requirements.txt
python main.py
# → http://0.0.0.0:8000 에서 서비스 시작
```

### 프론트엔드 개발 서버

```bash
cd prompt-improver-react/frontend
npm install
npm run dev
# → http://localhost:5173 에서 개발 서버 시작
```

### 프론트엔드 빌드 (프로덕션)

```bash
cd prompt-improver-react/frontend
npm run build
# → frontend/dist/ 에 빌드 결과물 생성
# → python main.py 실행 시 자동으로 dist/를 서빙
```

### 환경 변수 (.env)

```bash
GPT_API_BASE=http://내부서버주소/v1      # GPT API 엔드포인트
GPT_API_KEY=your-api-key                 # GPT API 키
GPT_MODEL=gpt-oss-120b-26               # 분석/Judge 모델명
DIFY_CLIENT_ID=your-client-id           # Dify 인증
DIFY_CLIENT_SECRET=your-client-secret   # Dify 인증
DIFY_BASE_URL=http://내부서버주소/v1      # Dify API 엔드포인트
```

## 3. 핵심 용어 정리

이 도구를 이해하기 위해 반드시 알아야 할 용어입니다.

| 용어 | 설명 |
|------|------|
| **Task (실험)** | 하나의 요약 유형(예: "민원내용", "불편사항")에 대한 최상위 실험 단위. 여러 Run을 포함 |
| **Run** | Task 내에서의 한 번의 실험 사이클. Phase 1~6까지 순차 실행 |
| **Zero 모드** | 처음부터 시작하는 Run. Judge JSON을 직접 업로드 |
| **Continue 모드** | 이전 Run의 결과를 이어받는 Run. Phase 4 결과가 자동으로 다음 Judge JSON이 됨 |
| **STT** | Speech-to-Text. 콜센터 상담 녹음의 음성인식 원본 텍스트 |
| **Reference** | 상담사가 수동으로 작성한 정답 요약. 이것이 "정답 기준" |
| **Generated** | LLM이 자동 생성한 요약. 이것의 품질을 개선하는 것이 목표 |
| **Keywords** | Reference에서 추출한 핵심 키워드. Judge가 판정 시 참조 |
| **Judge** | Generated를 Reference와 비교하여 정답/과답/오답을 판정하는 LLM |
| **Judge JSON** | Judge가 판정한 결과 파일. 각 케이스의 STT, Reference, Generated, 판정 결과를 포함 |
| **정답** | Generated가 Reference의 핵심 키워드를 정확히 포함 |
| **과답** | Reference 키워드를 포함하지만 불필요한 추가 내용도 있음 |
| **오답** | 핵심 키워드가 누락되거나, 잘못된 내용이 포함됨 |
| **버킷(Bucket)** | Phase 1에서 분류하는 오류 원인 유형 (STT오류, 프롬프트누락, 모델동작, Judge이견) |
| **Learning Rate** | 다음 Run에서 프롬프트를 얼마나 크게 바꿀지 자동 결정 (explore/major/medium/minor) |
| **Delta** | 이전 Run과 현재 Run의 케이스별 판정 변화 (개선/회귀/변화없음) |
| **앵커 가이드** | Phase 2에서 GPT가 항상 참조하는 고정 지침서. Task 단위로 설정 |
| **Mini-validation** | Phase 2에서 후보 프롬프트를 소수 케이스(3건)로 사전 검증하는 단계 |
| **Dify** | LLM 워크플로우 실행 플랫폼. 멀티노드 파이프라인을 시각적으로 구성 가능 |
| **노드(Node)** | Dify 워크플로우 내의 하나의 LLM 처리 단위. 최대 3개 노드로 구성 |
| **Reasoning** | GPT의 추론(thinking) 기능. High일수록 정확하지만 느림 |

## 4. 전체 파이프라인 흐름

```
실험(Task) 생성
 └─ Run 생성 (Zero 또는 Continue)
     ├─ Phase 1: 오류 원인 분석
     │    입력: Judge JSON + (선택) 현재 프롬프트
     │    출력: 케이스별 오류 원인(버킷), 전체 패턴 요약, 점수
     │
     ├─ Phase 2: 프롬프트 후보 설계
     │    입력: Phase 1 분석결과 + 실험 이력 + 앵커 가이드 + Phase 6 피드백
     │    출력: 개선 프롬프트 후보 3개 + Mini-validation 결과
     │
     ├─ Phase 3: Dify 워크플로우 실행
     │    입력: 선택된 후보 프롬프트 + 전체 STT 케이스
     │    출력: 전체 케이스의 새로운 Generated + 중간 노드 출력
     │
     ├─ Phase 4: Judge 재판정
     │    입력: 새 Generated + Reference + Keywords
     │    출력: 새 판정(정답/과답/오답) + 점수 + Delta
     │
     ├─ Phase 5: 성능 비교
     │    입력: 현재·이전 Run의 판정 결과
     │    출력: Delta 분석(개선/회귀/변화없음), 성능 트렌드 차트
     │
     └─ Phase 6: 전략 피드백
          입력: 프롬프트 변경점 + Delta + 중간 출력 + 실험 이력
          출력: 효과적 요소, 해로운 요소, 다음 방향, 제약사항
               └─ 다음 Run (Continue 모드)의 Phase 2에 자동 주입
```

**핵심 원리**: 각 Phase의 출력이 다음 Phase의 입력이 됩니다. Phase 6의 출력은 다시 다음 Run의 Phase 2에 주입되어, **반복할수록 GPT가 이전 실험에서 학습한 내용을 누적 반영**합니다.

## 5. 실험(Task) 생성

좌측 사이드바에서 **"+ 새 실험 만들기"** 버튼을 클릭합니다.

### 필수 입력

| 필드 | 설명 | 예시 |
|------|------|------|
| **Task명** | 실험 이름 | "민원내용 요약 개선", "불편사항 요약 v2" |

### 선택 입력

| 필드 | 설명 | 사용처 |
|------|------|--------|
| **설명** | 실험 목적이나 메모 | UI 표시용 |
| **요약 유형 (generation_task)** | 이 실험에서 다루는 요약 종류 | Phase 1~2에서 GPT가 "이것은 민원내용 요약 과제다"라고 이해하고 분석 방향을 맞춤 |

### 고급 설정 (접이식 패널)

#### LLM 설정 (분석 모델)

Phase 1 분석, Phase 2 설계, Phase 4 Judge, Phase 6 전략에 사용하는 GPT 모델을 지정합니다.
**비워두면 서버 기본값**(config.py의 `GPT_MODEL`)을 사용합니다.

| 필드 | 설명 | 기본값 |
|------|------|--------|
| API Base URL | OpenAI 호환 API 엔드포인트 | `GPT_API_BASE` 환경변수 |
| API Key | API 인증 키 | `GPT_API_KEY` 환경변수 |
| 모델명 | 모델 ID | `gpt-oss-120b-26` |

> **왜 Task별로 설정하나요?** 실험마다 다른 모델(예: 더 큰 모델, 다른 버전)을 사용하여 비교할 수 있습니다.

#### 시뮬레이션 LLM 설정 (생성 모델)

Phase 2 Mini-validation에서 워크플로우를 시뮬레이션할 때 사용하는 **생성 모델**입니다.
실제 Dify 워크플로우에서 사용하는 모델과 **동일한 모델**을 지정해야 Mini-validation 결과가 의미 있습니다.

| 필드 | 설명 | 예시 |
|------|------|------|
| API Base URL | 생성 모델 API 엔드포인트 | `http://생성모델서버/v1` |
| API Key | API 인증 키 | - |
| 모델명 | 모델 ID | `qwen3.5-35B-A3B` |

> **설정하지 않으면**: Mini-validation이 건너뛰어집니다. 후보 프롬프트의 사전 검증 없이 Phase 3으로 진행됩니다.

#### 앵커 가이드 (Phase 2 전략 고정 가이드)

GPT가 프롬프트를 설계할 때 **매 Run마다 항상 참조**하는 고정 지침서입니다.
`prompts/anchors/` 폴더의 `.txt` 파일 중 하나를 선택합니다.

> **사용 예시**: "민원내용" 요약에서 "고객의 핵심 요구 행위만 추출하라"는 원칙을 고정하면, GPT가 이 원칙을 벗어나지 않는 범위 내에서만 프롬프트를 변경합니다.

자세한 내용은 [15. 앵커 가이드](#15-앵커-가이드)를 참조하세요.

## 6. Run 생성

실험을 선택한 후, 상단 **"+ 새 Run"** 버튼을 클릭합니다.

### 시작 모드

| 모드 | 언제 사용 | 설명 |
|------|----------|------|
| **처음부터 (Zero)** | 첫 번째 Run, 또는 새 Judge JSON으로 시작할 때 | Judge JSON을 직접 업로드 |
| **이전 Run 이어서 (Continue)** | 2번째 Run부터 | 이전 Run의 Phase 4 판정 결과를 자동으로 Judge JSON으로 사용. Phase 6 피드백도 자동 주입 |

> **권장 워크플로우**: 첫 번째 Run은 **Zero**, 이후 Run은 **Continue**로 반복합니다.

### Continue 모드에서 자동으로 일어나는 일

Continue 모드로 Run을 생성하면 백엔드가 자동으로:

1. 이전 Run의 `case_results`를 Judge JSON 형식으로 변환하여 파일 생성
2. 이전 Run에서 선택한 후보의 프롬프트를 텍스트 파일로 추출
3. Phase 1에서 이전 Run의 중간 노드 출력(`intermediate_outputs`)을 분석 맥락에 포함
4. Phase 2에서 이전 Run의 Phase 6 피드백 체인을 최대 10단계까지 소급 수집하여 주입

## 7. Phase 1: 오류 원인 분석

### 무엇을 하나요?

업로드된 Judge JSON의 **오답/과답 케이스를 GPT가 하나씩 분석**하여, 왜 틀렸는지 원인을 분류(버킷)하고 개선 방향을 도출합니다.

### 사용 방법

1. **Judge JSON 업로드** (필수): 드래그 또는 클릭으로 업로드
2. **현재 프롬프트 업로드** (선택): 현재 사용 중인 요약 프롬프트 `.txt` 파일. 업로드하면 GPT가 "프롬프트의 어떤 지시가 누락/위반되었는지"까지 분석합니다.
3. **추론 수준(Reasoning) 선택** 후 **"Phase 1 실행"** 클릭

### 추론 수준 (Reasoning)

| 수준 | 설명 | 권장 상황 |
|------|------|----------|
| **High (정밀)** | GPT가 깊이 사고. 가장 정확한 분석 | 첫 Run, 오답률이 높을 때 (권장) |
| **Medium (균형)** | 속도와 정확도 균형 | 반복 실험 시 |
| **Low (빠름)** | 빠르지만 분석이 얕을 수 있음 | 빠른 확인이 필요할 때 |

### 입력 → 내부 동작 → 출력 (상세)

```
┌─ 입력 ──────────────────────────────────────────────────────────┐
│                                                                  │
│  1. Judge JSON 파일                                              │
│     각 케이스 구조:                                                │
│     {                                                            │
│       "id": "case_001",                                          │
│       "stt": "음성인식 원본 텍스트...",                             │
│       "reference": "상담사가 작성한 정답 요약",                      │
│       "keywords": "핵심,키워드,목록",                               │
│       "generated": "LLM이 생성한 요약",                            │
│       "answer_evaluation": "오답",  ← 필드명 자동 감지             │
│       "answer_evaluation_reason": "핵심 키워드 누락"                │
│     }                                                            │
│                                                                  │
│  2. (선택) 현재 프롬프트 TXT 파일                                   │
│                                                                  │
│  3. DB에서 읽는 데이터:                                             │
│     - tasks.generation_task   (어떤 요약 과제인지)                  │
│     - tasks.gpt_api_*         (GPT 모델 설정)                     │
│     - Continue 모드: 이전 Run의 intermediate_outputs               │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
                              ↓
┌─ 내부 동작 ─────────────────────────────────────────────────────┐
│                                                                  │
│  Step 1: JSON 파싱 및 분류                                        │
│    - 판정 필드 자동 감지 (evaluation/answer_evaluation/judge_result)│
│    - 정답 케이스: 즉시 DB 저장 (분석 불필요)                        │
│    - 오답/과답 케이스: GPT 분석 대상으로 분류                       │
│                                                                  │
│  Step 2: 케이스별 GPT 분석 (병렬, 동시 5건)                        │
│    GPT에 전달하는 컨텍스트:                                        │
│    - STT 원문, Reference(정답), Generated(LLM생성), Judge 판정     │
│    - 현재 프롬프트 텍스트 (업로드한 경우)                           │
│    - generation_task (요약 과제 설명)                               │
│    - 이전 Run의 중간 노드 출력 (Continue 모드)                      │
│                                                                  │
│    GPT가 분석하여 반환하는 항목:                                    │
│    - bucket: 오류 원인 분류                                       │
│      · stt_error: STT 오전사가 원인 (프롬프트로 해결 불가)          │
│      · prompt_missing: 프롬프트에 지시가 없어서 발생                │
│      · model_behavior: 지시는 있지만 모델이 따르지 않음             │
│      · judge_dispute: 사람이 보면 맞는데 Judge가 틀렸다고 판정      │
│    - analysis_summary: 오류 원인 요약                              │
│    - missing_instruction: 프롬프트에 누락된 지시사항                │
│    - violated_instruction: 위반된 기존 지시사항                     │
│    - error_pattern: 오류 패턴 이름                                 │
│    - improvement_suggestion: 개선 제안                             │
│    - reference_criteria: 상담사(Reference)의 요약 기준              │
│    - content_gap: Generated와 Reference의 내용 차이                │
│    - stt_uncertain_expressions: STT에서 의미 불명확한 부분          │
│    - hallucination_detected: LLM이 없는 내용을 만들어냈는지         │
│    - judge_agreement/dispute_reason: Judge 판정 동의 여부           │
│                                                                  │
│  Step 3: 전체 패턴 요약 (GPT 1회 호출)                             │
│    모든 케이스 분석을 종합하여:                                     │
│    - top_issues: 가장 빈번한 오류 이슈 3가지                       │
│    - recommended_focus: Phase 2에서 집중해야 할 방향                │
│    - reference_summary_criteria: 상담사가 어떤 기준으로 요약하는지  │
│    - common_content_gaps: 공통적인 Generated-Reference 차이 패턴   │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
                              ↓
┌─ 출력 ──────────────────────────────────────────────────────────┐
│                                                                  │
│  1. DB 저장 (case_results 테이블):                                │
│     - 전체 케이스: stt, reference, keywords, generated, evaluation│
│     - 오답/과답 케이스: bucket, analysis_summary, 모든 분석 필드   │
│                                                                  │
│  2. DB 저장 (phase_results.output_data):                          │
│     {                                                            │
│       "scores": {                                                │
│         "correct_plus_over": 87.5,  ← 핵심 지표                  │
│         "correct": 80.0, "over": 7.5, "wrong": 12.5             │
│       },                                                         │
│       "bucket_counts": {                                         │
│         "stt_error": 2, "prompt_missing": 8,                    │
│         "model_behavior": 3, "judge_dispute": 0                 │
│       },                                                         │
│       "top_issues": ["이슈1", "이슈2", "이슈3"],                  │
│       "recommended_focus": "Phase 2 방향 제안",                   │
│       "reference_summary_criteria": "상담사 요약 기준 패턴",       │
│       "error_pattern_ranking": [{"pattern":"..","count":N}],     │
│       "prompt_improvable_cases": [케이스 상세 목록]                │
│     }                                                            │
│                                                                  │
│  3. 화면에 표시되는 것:                                            │
│     - 점수 카드 4개, 판정 분포 차트, 버킷 분포 차트                 │
│     - 케이스 테이블 (클릭 시 상세 분석)                             │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 화면 설명

#### 점수 카드 (4개)

| 카드 | 의미 |
|------|------|
| **정답+과답%** | 핵심 지표. 이 비율을 95% 이상으로 올리는 것이 목표 |
| **정답%** | Reference와 동일한 핵심 내용을 요약한 비율 |
| **과답%** | 맞지만 불필요한 내용이 추가된 비율 |
| **오답%** | 핵심 내용이 누락되거나 틀린 비율 |

#### 판정 분포 차트

- **왼쪽**: 정답/과답/오답 건수 막대 그래프
- **오른쪽**: 오류 원인 분류(버킷) 분포

#### 버킷 설명

| 버킷 | 색상 | 의미 | 프롬프트로 해결 가능? |
|------|------|------|---------------------|
| **STT 오류** | 분홍 | STT 음성인식 자체의 오전사가 원인 | **불가** |
| **프롬프트 누락** | 노랑 | 프롬프트에 해당 지시가 없어서 발생 | **가능** (핵심 개선 대상) |
| **모델 동작** | 파랑 | 프롬프트 지시는 있지만 모델이 따르지 않음 | 부분적 (강조/구조 변경으로 시도) |
| **Judge 이견** | 보라 | 사람이 보면 맞는데 Judge가 오답으로 판정 | 불필요 (Judge 기준 문제) |

> **핵심 인사이트**: "프롬프트 누락" 비율이 높을수록 Phase 2에서 개선 여지가 큽니다.

#### 케이스 테이블

| 컬럼 | 설명 |
|------|------|
| ID | 케이스 고유 ID |
| Judge판정 | 정답(녹색)/과답(노란색)/오답(빨간색) |
| 버킷 | 오류 원인 분류 |
| 분석사유 | GPT가 분석한 오류 원인 요약 |
| STT불확실표현 | STT에서 의미가 불명확한 부분 |
| STT | 원본 음성인식 텍스트 |
| Reference | 상담사가 작성한 정답 요약 |
| Generated | LLM이 생성한 요약 |
| Judge이견 | Judge 판정에 의문이 있는 경우 이유 |

각 행을 클릭하면 **상세 분석** (누락 지시, 위반 지시, 오류 패턴, 개선 제안 등)을 확인할 수 있습니다.

> **다운로드**: "JSON ⬇" 또는 "XLSX ⬇" 버튼으로 분석 결과를 내려받을 수 있습니다.

## 8. Phase 2: 프롬프트 후보 설계

### 무엇을 하나요?

Phase 1 분석 결과를 바탕으로 GPT가 **개선된 프롬프트 후보 3개**를 자동 설계합니다.
시뮬레이션 모델이 설정된 경우, Mini-validation으로 후보를 사전 검증합니다.

### 사용 방법

1. (선택) **전략 가이드** 입력: 사용자가 직접 설계 방향을 지시할 수 있습니다.
   예: "Chain of thought 사용 금지", "1노드로 한정", "키워드 체크리스트를 반드시 포함"
2. **추론 수준 선택** 후 **"Phase 2 실행"** 클릭
3. 생성된 후보 카드에서 **"후보 선택"** 클릭, 또는 **커스텀 후보** 직접 작성

### 입력 → 내부 동작 → 출력 (상세)

```
┌─ 입력 (GPT에 전달되는 전체 컨텍스트) ──────────────────────────┐
│                                                                  │
│  Phase 1 결과 (DB: phase_results[phase=1].output_data):          │
│    - bucket_counts: 오류 원인별 분포                               │
│    - error_pattern_ranking: 빈도순 오류 패턴                       │
│    - top_issues: 주요 이슈 3가지                                   │
│    - recommended_focus: 개선 방향 제안                              │
│    - reference_summary_criteria: 상담사 요약 기준                   │
│    - prompt_improvable_cases: 프롬프트로 개선 가능한 케이스 상세     │
│    - scores: 현재 점수                                             │
│                                                                  │
│  실험 이력 (DB: 이전 runs + prompt_candidates + phase_results):     │
│    - 각 Run의 점수, 선택된 후보, 노드 구조, 프롬프트 텍스트          │
│    - "Run 1: explore/67.5% → Run 2: converge/81.2% → ..."        │
│                                                                  │
│  Learning Rate (services/delta.py에서 자동 계산):                  │
│    - explore: 첫 Run → 자유 탐색                                  │
│    - major: 점수 < 70% 또는 3 Run 연속 정체 → 구조 대폭 변경       │
│    - medium: 점수 70~85% → 요소 재조합                             │
│    - minor: 점수 ≥ 85% → 미세 조정만                               │
│                                                                  │
│  Converge 컨텍스트 (2번째 Run부터):                                │
│    - 이전 최고 성적 프롬프트의 전체 구조와 텍스트                     │
│    - "이 프롬프트를 기반으로 점진 개선하라"                          │
│                                                                  │
│  Phase 6 피드백 체인 (Continue 모드):                               │
│    - 이전 Run의 Phase 6 결과 (effective/harmful/next_direction)    │
│    - 최대 10단계 ancestor까지 소급 수집                             │
│                                                                  │
│  사용자 전략 가이드 (입력한 경우):                                   │
│    - 사용자가 직접 입력한 설계 방향 지시                             │
│                                                                  │
│  앵커 가이드 (Task에 설정된 경우):                                   │
│    - prompts/anchors/*.txt 파일의 고정 지침                        │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
                              ↓
┌─ 내부 동작 (4단계 파이프라인) ───────────────────────────────────┐
│                                                                  │
│  Step 1/4: 전략 수립 (GPT 1회 호출)                               │
│    위의 모든 컨텍스트를 종합하여 3가지 서로 다른 전략 방향 설계      │
│    각 전략: 노드 수, 노드별 역할 분담, reasoning 설정, 집중 패턴    │
│    출력: candidates 배열 + design_summary                         │
│                                                                  │
│  Step 2/4: 후보별 프롬프트 생성 (GPT 3회, 병렬 호출)               │
│    각 전략에 맞는 구체적인 system_prompt + user_prompt 작성          │
│    대표 오류 케이스(최대 5건)를 참조하여 동일 오류 방지              │
│    출력: 노드별 system_prompt, user_prompt, 입출력 변수              │
│                                                                  │
│  Step 3/4: 검증 및 보정                                           │
│    누락된 노드가 있으면 GPT로 자동 보정 (repair 호출)               │
│                                                                  │
│  Step 4/4: Mini-validation (시뮬레이션 LLM 설정 시에만)            │
│    1) Phase 1에서 발견된 오답 중 3건을 선별                         │
│    2) 각 후보의 워크플로우를 시뮬레이션 모델로 실제 실행             │
│       (노드별 순차: Node A 출력 → Node B 입력 → ... → generated)   │
│    3) Judge 프롬프트로 결과를 판정 → pass_rate 계산                 │
│    4) pass_rate > 0인 후보만 유지                                  │
│    5) 모든 후보 실패 시:                                           │
│       - 실패 피드백을 GPT에 전달하여 프롬프트 재생성 (1회 재시도)    │
│       - 재시도에서도 실패 시: 최선 1개를 추천                       │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
                              ↓
┌─ 출력 ──────────────────────────────────────────────────────────┐
│                                                                  │
│  1. DB 저장 (prompt_candidates 테이블):                           │
│     - 후보별 (A, B, C): mode, node_count, design_rationale       │
│     - 노드별: system_prompt, user_prompt, input_vars, output_var  │
│     - 노드별: model, reasoning 설정                               │
│                                                                  │
│  2. DB 저장 (phase_results.output_data):                          │
│     {                                                            │
│       "mode": "explore | converge",                              │
│       "learning_rate": "explore | major | medium | minor",        │
│       "design_summary": "전체 설계 방향 요약",                     │
│       "candidates": [후보 상세 배열],                              │
│       "mini_validation_summary": {                                │
│         "enabled": true,                                         │
│         "validation_case_count": 3,                              │
│         "candidate_results": [                                   │
│           {                                                      │
│             "label": "A", "pass_rate": 0.67,                     │
│             "passed": 2, "total": 3,                             │
│             "details": [케이스별 판정+생성결과]                     │
│           }                                                      │
│         ]                                                        │
│       }                                                          │
│     }                                                            │
│                                                                  │
│  3. 화면에 표시되는 것:                                            │
│     - 후보 카드 (A, B, C), 커스텀 편집기                           │
│     - Mini-Validation 결과 패널, 디자인 요약                       │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### Design Mode

| 모드 | 조건 | GPT의 행동 |
|------|------|-----------|
| **Explore** | 첫 번째 Run | 완전히 다른 3가지 구조를 실험. 1노드/2노드/3노드 등 다양한 조합 |
| **Converge** | 2번째 Run~ | 이전 최고 성적 프롬프트를 기반으로 점진 개선. 구조는 유지하되 지시사항 보완 |

### Learning Rate (자동 계산)

| 값 | 조건 | Phase 2에서의 의미 |
|------|------|-------------------|
| **explore** | 완료된 Run 없음 | 자유 탐색, 다양한 구조 실험 |
| **major** | 점수 < 70% 또는 3 Run 연속 정체(편차 < 2%) | 대폭 구조 변경 |
| **medium** | 점수 70~85% | 요소 재조합, 중간 수준 변경 |
| **minor** | 점수 ≥ 85% | 미세 조정만 (표현 수정, 예시 추가 등) |

### 화면 설명

#### 후보 카드

각 후보(A, B, C)의 카드에 표시되는 정보:

| 영역 | 설명 |
|------|------|
| **헤더** | 후보 라벨 + 노드 수 + 모드(explore/converge) |
| **설계 근거** | GPT가 이 전략을 선택한 이유 |
| **노드 탭** | 멀티노드인 경우 Node A / Node B / Node C 탭 전환 |
| **System / User 탭** | 각 노드의 시스템 프롬프트와 사용자 프롬프트 |
| **입출력 변수** | 각 노드가 받는 입력과 생성하는 출력 변수 |
| **"후보 선택" 버튼** | 이 후보를 Phase 3에서 사용할 프롬프트로 선택 |
| **"복사" 버튼** | 이 후보의 프롬프트를 커스텀 편집기에 복사 |

#### Mini-Validation 결과 패널

시뮬레이션 LLM이 설정된 경우에만 표시됩니다.

| 영역 | 설명 |
|------|------|
| **후보별 그룹** | 각 후보의 pass_rate와 통과 건수. pass_rate에 따라 초록(≥67%)/노랑(34~66%)/빨강(<34%) 배경 |
| **케이스 테이블** | 각 검증 케이스의 판정(정답/과답/오답)과 생성된 결과 미리보기 |
| **행 클릭** | 생성 결과 전체(최대 200자)를 펼쳐서 확인 |

#### 커스텀 후보 작성

GPT 생성 후보를 수정하거나 완전히 새로운 프롬프트를 작성할 수 있습니다.

| 설정 | 설명 |
|------|------|
| **노드 수** | 1~3개. 단일 노드로 충분하면 1개로 |
| **Reasoning** | 체크하면 해당 노드에서 GPT가 추론(thinking) 후 응답 |
| **System Prompt** | 역할 정의, 규칙, 제약조건 |
| **User Prompt** | 실제 데이터 처리 템플릿. `{stt}` 등 변수 사용 |
| **입력 변수** | 이 노드가 필요로 하는 변수 (첫 노드는 반드시 `stt` 포함) |
| **출력 변수** | 이 노드가 생성하는 변수 (마지막 노드는 반드시 `generated`) |

> **노드 구조 예시**: Node A가 STT에서 핵심 정보를 추출(→ `analysis`), Node B가 `analysis`를 바탕으로 최종 요약 생성(→ `generated`)

## 9. Phase 3: Dify 워크플로우 실행

### 무엇을 하나요?

Phase 2에서 선택한 프롬프트를 **Dify 워크플로우에서 실행**하여, 전체 케이스의 새로운 요약(`generated`)을 생성합니다.

### 사전 준비: Dify 워크플로우 구성

Phase 3을 실행하기 전에 Dify 플랫폼에서 워크플로우를 구성해야 합니다:

1. Phase 2에서 설계된 **노드 구조**를 Dify에 동일하게 구현
2. 각 노드의 **System Prompt, User Prompt**를 복사
3. **입출력 변수**를 동일하게 설정 (첫 노드 입력: `stt`, 마지막 노드 출력: `generated`)
4. Dify에서 워크플로우의 **Object ID**를 확인

### 사용 방법

1. **선택된 후보 확인**: 상단에 선택된 후보의 구조와 프롬프트가 표시됩니다
2. **Dify Workflow Object ID 입력**: Dify에서 확인한 워크플로우 ID를 입력
3. **"연결 확인"** 클릭: 워크플로우가 정상 동작하는지 테스트 호출
4. 연결 확인 성공 후 **"전체 케이스 실행"** 클릭

### 입력 → 내부 동작 → 출력 (상세)

```
┌─ 입력 ──────────────────────────────────────────────────────────┐
│                                                                  │
│  1. Dify Object ID (사용자 입력)                                  │
│  2. DB에서 읽는 데이터:                                           │
│     - case_results: 전체 케이스의 case_id, stt                    │
│     - prompt_candidates: 선택된 후보의 노드별 output_var           │
│       (중간 출력 변수명 매핑에 사용)                                │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
                              ↓
┌─ 내부 동작 ─────────────────────────────────────────────────────┐
│                                                                  │
│  1. Dify 토큰 발급 (dify_auth.py)                                │
│     DIFY_CLIENT_ID + DIFY_CLIENT_SECRET → Bearer Token           │
│                                                                  │
│  2. 케이스별 Dify API 호출 (병렬, 동시 5건)                       │
│     POST {DIFY_BASE_URL}/workflows/run                           │
│     입력: {"inputs": {"stt": "STT 전문"}}                        │
│     타임아웃: 120초, 실패 시 최대 3회 재시도                       │
│                                                                  │
│  3. 응답 파싱                                                     │
│     Dify 응답의 data.outputs에서:                                 │
│     - "generated" 키 → 최종 요약 결과                             │
│     - 기타 키(analysis, keywords 등) → 중간 노드 출력             │
│     - output_var와 노드 라벨 매핑하여 intermediate_outputs 구성    │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
                              ↓
┌─ 출력 ──────────────────────────────────────────────────────────┐
│                                                                  │
│  DB 저장 (case_results 테이블, 케이스별 UPDATE):                  │
│    - generated: 최종 요약 결과 텍스트                              │
│    - intermediate_outputs (JSON):                                │
│      {                                                           │
│        "analysis": {"node": "A", "content": "중간 분석..."},     │
│        "keywords": {"node": "B", "content": "추출 키워드..."}    │
│      }                                                           │
│                                                                  │
│  화면: 진행 바 + 완료/오류 카운트                                  │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

| 영역 | 설명 |
|------|------|

| **Dify 연결** | Object ID 입력 + 연결 확인 상태 (✓ 녹색 / ✕ 빨간색) |

## 10. Phase 4: Judge 재판정

### 무엇을 하나요?

Phase 3에서 새로 생성된 요약을 **GPT Judge가 다시 판정**합니다.
이전 Run과 동일한 Judge 기준으로 평가하여, 프롬프트 변경 효과를 객관적으로 측정합니다.

### 사용 방법

**"Phase 4 실행"** 클릭만 하면 됩니다. 별도 입력이 필요 없습니다.

### 입력 → 내부 동작 → 출력 (상세)

```
┌─ 입력 ──────────────────────────────────────────────────────────┐
│                                                                  │
│  1. DB에서 읽는 데이터:                                           │
│     - case_results: generated가 있는 케이스의                     │
│       case_id, stt, generation_task, reference, keywords,        │
│       generated                                                  │
│     - tasks.gpt_api_* (GPT 설정)                                 │
│                                                                  │
│  2. 프롬프트 파일:                                                │
│     - prompts/phase4_judge.txt (시스템 프롬프트, 수동 관리)       │
│     - prompts/phase4_judge_user.txt (유저 프롬프트 템플릿)        │
│       → 변수: {stt}, {generation_task}, {reference},              │
│               {keywords}, {generated}                            │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
                              ↓
┌─ 내부 동작 ─────────────────────────────────────────────────────┐
│                                                                  │
│  1. 케이스별 Judge GPT 호출 (병렬, 동시 5건, reasoning=low 고정)  │
│     GPT에 전달:                                                   │
│     - System: Judge 시스템 프롬프트 (판정 기준, 출력 형식)         │
│     - User: STT + 요약과제 + Reference + Keywords + Generated    │
│                                                                  │
│  2. 응답 파싱 (우선순위):                                         │
│     ① JSON 파싱: {"rating": "정답|과답|오답", "reason": "..."}    │
│     ② JSON 파싱 실패 시 텍스트 분류:                              │
│        - "결론 상반" 감지 → 오답                                  │
│        - 키워드 매칭 (한국어: 오답/과답/정답 → 영어: wrong/extra)  │
│        - 매칭 실패 → 평가실패                                     │
│                                                                  │
│  3. 점수 집계                                                     │
│     정답%, 과답%, 오답%, 정답+과답%, 총 건수                       │
│                                                                  │
│  4. Delta 계산 (이전 Run 존재 시)                                  │
│     이전 Run의 case_results와 비교:                                │
│     - improved: 오답 → 정답/과답                                  │
│     - regressed: 정답/과답 → 오답                                 │
│     - unchanged: 동일 판정 또는 양쪽 다 positive(정답↔과답)       │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
                              ↓
┌─ 출력 ──────────────────────────────────────────────────────────┐
│                                                                  │
│  1. DB 저장 (case_results): evaluation, reason 업데이트           │
│  2. DB 저장 (case_deltas): 케이스별 변화 기록                     │
│  3. DB 저장 (runs): score_correct, score_over, score_total        │
│  4. DB 저장 (phase_results.output_data):                          │
│     {                                                            │
│       "scores": {                                                │
│         "correct_plus_over": 87.5,                               │
│         "correct": 80.0, "over": 7.5, "wrong": 12.5,            │
│         "total": 100                                             │
│       },                                                         │
│       "cases": [케이스별 판정+사유+중간출력 전체 배열]              │
│     }                                                            │
│                                                                  │
│  화면: 점수 카드 4개 + 케이스 테이블                               │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 화면 설명

| 영역 | 설명 |
|------|------|
| **점수 카드** | Phase 1과 동일한 4개 카드. 새로운 프롬프트의 성적 |
| **케이스 테이블** | 각 케이스의 판정, 사유, STT, Reference, Generated |

> **행 클릭 시**: 중간 출력(각 노드가 생성한 중간 결과)까지 확인할 수 있습니다. 이를 통해 어떤 노드에서 문제가 발생했는지 디버깅할 수 있습니다.

## 11. Phase 5: 성능 비교 (Delta 분석)

### 무엇을 하나요?

Phase 4 결과를 **이전 Run과 비교**하여, 프롬프트 변경이 실제로 효과가 있었는지 분석합니다.

### 사용 방법

**별도 실행 버튼이 없습니다.** Phase 4 완료 시 자동으로 계산됩니다.

### 입력 → 출력 (상세)

```
┌─ 입력 ──────────────────────────────────────────────────────────┐
│                                                                  │
│  DB에서 읽는 데이터:                                              │
│  - case_deltas: Phase 4에서 계산된 Delta (improved/regressed/    │
│    unchanged)                                                    │
│  - case_results (현재 run): 전체 케이스 판정 + 중간 출력          │
│  - case_results (이전 run): 이전 Generated (비교용)               │
│  - runs (task 전체): 모든 Run의 점수 → 트렌드 차트                │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
                              ↓
┌─ 출력 (REST API 응답, SSE 아님) ─────────────────────────────────┐
│                                                                  │
│  {                                                               │
│    "scores": { 현재 Run 점수 },                                   │
│    "delta": {                                                    │
│      "improve": 8,    ← 오답→정답/과답 전환 건수                  │
│      "regress": 3,    ← 정답/과답→오답 전환 건수                  │
│      "same": 89       ← 변화 없는 건수                            │
│    },                                                            │
│    "trend": {                                                    │
│      "labels": ["Run 1", "Run 2", "Run 3"],                     │
│      "values": [72.0, 81.5, 87.5]   ← 각 Run의 정답+과답%       │
│    },                                                            │
│    "regressed_cases": [회귀 케이스 상세 목록],                     │
│    "goal_achieved": false,                                       │
│    "gap_to_goal": 7.5,   ← 95% - 현재 점수                      │
│    "cases": [전체 케이스 + delta_type + prev_judge + prev_generated]│
│  }                                                               │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 화면 설명

#### 목표 달성 배너

- **녹색**: "목표 달성!" (정답+과답 ≥ 95%)
- **노란색**: "목표까지 X%p 부족"

#### Run별 성능 추이 차트

모든 Run의 정답+과답% 변화를 라인 차트로 보여줍니다.
> 첫 번째 Run에서는 "다음 Run부터 추이 비교가 가능합니다" 메시지가 표시됩니다.

#### Delta 카드 (3개)

| 카드 | 의미 |
|------|------|
| **개선** (녹색) | 이전에 오답이었는데 이번에 정답/과답으로 바뀐 케이스 수 |
| **회귀** (빨간색) | 이전에 정답/과답이었는데 이번에 오답으로 바뀐 케이스 수 |
| **변화없음** (회색) | 판정이 동일한 케이스 수 |

> **핵심**: **개선 > 회귀**여야 프롬프트 변경이 효과적입니다. 회귀가 더 많으면 프롬프트가 오히려 나빠진 것입니다.

#### 회귀 케이스 테이블

회귀된 케이스만 모아서 보여줍니다. 이전 판정 → 현재 판정과 Judge 사유를 확인하여 **왜 나빠졌는지** 파악하세요.

#### 전체 케이스 테이블

| 변화 배지 | 의미 |
|-----------|------|
| **개선** (녹색) | 오답 → 정답/과답 |
| **회귀** (빨간색) | 정답/과답 → 오답 |
| **변화없음** (회색) | 동일 |
| **신규** (파란색) | 이전 Run에 없던 케이스 |

행 클릭 시 **이전 Run의 Generated와 현재 Generated를 나란히 비교**할 수 있습니다.

## 12. Phase 6: 전략 피드백

### 무엇을 하나요?

이번 Run의 결과를 종합 분석하여, **다음 Run에서 무엇을 바꿔야 하는지** GPT가 피드백을 생성합니다.
이 피드백은 다음 Run (Continue 모드)의 Phase 2에 자동으로 주입됩니다.

### 사용 방법

**"Phase 6 실행"** 클릭

### 입력 → 내부 동작 → 출력 (상세)

```
┌─ 입력 (GPT에 전달되는 전체 컨텍스트) ──────────────────────────┐
│                                                                  │
│  1. 프롬프트 변경 비교:                                           │
│     - 현재 Run에서 선택한 후보의 전체 프롬프트 구조                 │
│     - 이전 Run에서 선택한 후보의 전체 프롬프트 구조                 │
│     - 양자 간 diff (노드 수, reasoning 설정, 프롬프트 텍스트 변경점)│
│                                                                  │
│  2. Delta 분석 데이터:                                            │
│     - case_deltas: 전체 변화 기록 (최대 50건)                      │
│     - 개선 케이스: 어떤 케이스가 왜 좋아졌는지                     │
│     - 회귀 케이스 상세 (최대 10건):                                │
│       현재 Judge 사유 + 이전/현재 중간 노드 출력 비교              │
│                                                                  │
│  3. 오류 케이스 중간 출력 (최대 5건):                              │
│     현재 오답/과답 케이스의 intermediate_outputs                   │
│     → 어떤 노드에서 문제가 발생했는지 분석 재료                    │
│                                                                  │
│  4. Phase 1의 Reference 요약 기준:                                │
│     reference_summary_criteria (상담사가 어떻게 요약하는지)        │
│                                                                  │
│  5. 실험 이력:                                                    │
│     전체 Task의 Run별 점수, 프롬프트 변경 이력                     │
│                                                                  │
│  6. Learning Rate:                                                │
│     현재 점수 추이에 따라 자동 계산                                 │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
                              ↓
┌─ 내부 동작 ─────────────────────────────────────────────────────┐
│                                                                  │
│  GPT 1회 호출 (reasoning=high):                                   │
│  위의 모든 컨텍스트를 종합하여:                                    │
│                                                                  │
│  1. Backprop 분석                                                │
│     "프롬프트의 어떤 변경이 어떤 케이스의 판정 변화를 유발했는가"   │
│     (Neural network의 backpropagation에서 착안한 인과 분석)       │
│                                                                  │
│  2. 효과적 요소 식별                                              │
│     "이번 변경 중 유지해야 할 것" (개선 케이스와 연관된 변경)       │
│                                                                  │
│  3. 해로운 요소 식별                                              │
│     "이번 변경 중 제거해야 할 것" (회귀 케이스와 연관된 변경)       │
│                                                                  │
│  4. 다음 방향 제안                                                │
│     구체적인 개선 방향 (무엇을 추가/수정/삭제할지)                  │
│                                                                  │
│  5. 제약사항 식별                                                 │
│     절대 바꾸지 말아야 할 요소                                     │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
                              ↓
┌─ 출력 ──────────────────────────────────────────────────────────┐
│                                                                  │
│  DB 저장 (phase_results.output_data):                             │
│  {                                                               │
│    "learning_rate": "minor",                                     │
│    "backprop": "프롬프트 변경 인과 분석 텍스트...",                 │
│    "effective": ["효과적 요소1", "효과적 요소2"],                  │
│    "harmful": ["해로운 요소1"],                                    │
│    "next_direction": "다음 Run 개선 방향 텍스트...",               │
│    "constraints": "반드시 유지할 요소 텍스트..."                   │
│  }                                                               │
│                                                                  │
│  → 다음 Run (Continue 모드)의 Phase 2에 자동 주입됨               │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 화면 설명

| 영역 | 설명 |
|------|------|
| **Learning Rate** | 파란 배지. 다음 Run에서의 변경 폭을 나타냄 |
| **Backprop 분석** | 프롬프트 변경 → 케이스별 영향 인과 분석 텍스트 |
| **효과적 요소** (녹색 태그) | 다음 Run에서 유지해야 할 변경점 |
| **해로운 요소** (빨간 태그) | 다음 Run에서 제거해야 할 변경점 |
| **다음 방향** | 다음 Run의 구체적 개선 방향 |

> Phase 6 완료 후 **"다음 Run 시작 →"** 버튼이 나타납니다.

---

## 13. 다음 Run 시작 (반복 개선)

Phase 6 완료 후 **"다음 Run 시작 →"** 클릭 시:

1. **Continue 모드**로 새 Run 자동 생성
2. Phase 4 판정 결과가 자동으로 다음 Run의 Judge JSON이 됨
3. Phase 6 피드백이 Phase 2에 자동 주입
4. 누적된 효과/해로운 패턴, 제약사항이 모두 반영

```
Run 1 (Zero)                    Run 2 (Continue)              Run 3 (Continue)
Phase 1: 72.0%                  Phase 1: 81.5%                Phase 1: 87.5%
Phase 2: explore 모드            Phase 2: converge 모드         Phase 2: converge 모드
  └ 자유 구조 탐색                 └ Run 1 최고 기반 개선         └ Run 2 최고 기반 개선
  └ learning_rate: explore        └ learning_rate: medium        └ learning_rate: minor
                                   └ Phase 6 피드백 주입          └ Phase 6 피드백 체인
Phase 6: 효과적/해로운 분석 ───→  Phase 6: 누적 분석 ──────→   Phase 6: ...
```

> **반복할수록**: GPT가 이전 실험에서 학습한 내용을 **누적** 반영합니다. Phase 6 피드백은 최대 10단계 ancestor까지 소급 수집되어, 오래된 실험의 인사이트도 활용됩니다.

## 14. Continue 모드 상세: 데이터 자동 이관

Continue 모드에서 어떤 데이터가 어떻게 이관되는지 상세히 설명합니다.

### Judge JSON 자동 생성

이전 Run의 `case_results`를 다음 형식으로 변환:

```json
[
  {
    "id": "case_001",
    "generation_task": "민원내용",
    "stt": "음성인식 원문...",
    "reference": "정답 요약",
    "keywords": "핵심,키워드",
    "generated": "이전 Run에서 LLM이 생성한 요약",
    "answer_evaluation": "오답",
    "answer_evaluation_reason": "키워드 누락..."
  }
]
```

### 프롬프트 자동 이관

이전 Run에서 선택한 후보의 프롬프트를 텍스트 파일로 추출하여, Phase 1에서 "현재 프롬프트"로 참조합니다.

### Phase 1에서의 추가 컨텍스트

Continue 모드의 Phase 1에서는 이전 Run의 `intermediate_outputs` (중간 노드 출력)도 분석 컨텍스트에 포함됩니다. 이를 통해 GPT가 "Node A에서 정보 추출은 잘 되었지만 Node B에서 요약 과정에서 누락되었다"와 같은 **노드 수준 분석**을 수행할 수 있습니다.

### Phase 2에서의 피드백 체인

Phase 2는 이전 Run의 Phase 6 피드백뿐만 아니라, 그 이전 Run의 Phase 6까지 최대 **10단계 ancestor를 소급**하여 피드백 체인을 구성합니다:

```
Run 5의 Phase 2가 참조하는 피드백:
  ← Run 4 Phase 6: "키워드 체크리스트 추가가 효과적이었음"
  ← Run 3 Phase 6: "2노드 구조가 1노드보다 안정적"
  ← Run 2 Phase 6: "reasoning=true가 정확도를 높임"
  ...
```

## 15. 앵커 가이드

### 앵커 가이드란?

앵커 가이드는 `prompts/anchors/` 디렉토리에 저장된 `.txt` 파일로, Phase 2에서 GPT가 프롬프트를 설계할 때 **항상 참조**하는 고정 지침서입니다.

### 왜 필요한가?

GPT가 프롬프트를 자동 설계할 때, 요약 과제별로 반드시 지켜야 할 원칙이 있습니다. 예를 들어:

- "민원내용" 요약: 고객의 핵심 요구 행위만 추출, 상담 경위 제외
- "불편사항" 요약: 감정 키워드(불만, 짜증) 반드시 포함
- "전체상담요약": 상담사의 최종 안내/결론이 가장 중요

이런 원칙은 매 Run마다 변하지 않으므로, 앵커 가이드로 고정합니다.

### 구조 (6개 섹션)

| 섹션 | 내용 |
|------|------|
| **1. 요약 대상의 본질** | 이 요약이 무엇인지, 어디에 사용되는지 |
| **2. 요약 구조** | 출력 형태, Reference 예시 |
| **3. 추출 우선순위** | STT에서 무엇을 뽑아야 하는지 (중요도 순서) |
| **4. 금지 사항** | 하면 안 되는 것 (키워드 변형, 과답, 문체 등) |
| **5. STT 오전사 대응** | 음성인식 오류 처리 방법 |
| **6. 자주 틀리는 패턴과 교정** | 오답/정답 예시 쌍 |

### 현재 제공되는 앵커 가이드

| 파일명 | 요약 과제 |
|--------|----------|
| `민원내용.txt` | 민원내용 요약 |
| `요구사항.txt` | 요구사항 요약 |
| `전체상담요약.txt` | 상담내용요약 (전체상담) |
| `불편사항.txt` | 불편사항 요약 |
| `개선요청사항.txt` | 개선요청사항 요약 |

### 새 앵커 가이드 만들기

1. LLM Judge 결과 파일(JSON/XLSX)을 분석하여 오답 패턴 파악
2. 위의 6개 섹션 구조에 맞춰 `.txt` 파일 작성
3. `prompts/anchors/` 디렉토리에 저장
4. Task 설정에서 해당 파일 선택

## 18. FAQ / 트러블슈팅

### Q: Phase 2에서 Mini-validation이 실행되지 않아요

**A**: Task 설정에서 **시뮬레이션 LLM 설정**(API Base URL, API Key, 모델명)을 모두 입력해야 합니다. 비어있으면 Mini-validation이 건너뛰어집니다.

### Q: Phase 3에서 Dify 연결 확인이 실패해요

**A**: 다음을 확인하세요:
1. `DIFY_CLIENT_ID`와 `DIFY_CLIENT_SECRET`이 올바른지
2. `DIFY_BASE_URL`이 접근 가능한 주소인지
3. Dify 워크플로우가 "게시됨" 상태인지
4. Object ID가 정확한지

### Q: Judge JSON 파일의 형식이 어떻게 되어야 하나요?

**A**: JSON 배열로, 각 객체에 최소한 다음 필드가 필요합니다:

```json
[
  {
    "id": "고유ID",
    "stt": "음성인식 텍스트",
    "reference": "정답 요약",
    "generated": "LLM 생성 요약",
    "answer_evaluation": "정답|과답|오답"
  }
]
```

판정 필드명은 자동 감지됩니다: `evaluation`, `answer_evaluation`, `judge_result`, `judge`, `result` 중 하나.
`keywords` 필드가 있으면 Judge가 키워드 기반 판정을 수행합니다.

### Q: 페이지를 새로고침하면 데이터가 사라져요

**A**: 데이터는 DB에 저장되어 있으므로 사라지지 않습니다. 새로고침 시 자동으로 복원됩니다. 만약 복원이 안 된다면 브라우저 콘솔에서 에러를 확인하세요.

### Q: Continue 모드와 Zero 모드의 차이가 뭔가요?

**A**:
- **Zero**: 새 Judge JSON을 직접 업로드합니다. 이전 Run과 무관하게 시작합니다.
- **Continue**: 이전 Run의 Phase 4 판정 결과가 자동으로 Judge JSON이 됩니다. Phase 6 피드백도 자동 주입되어, GPT가 이전 실험의 인사이트를 반영합니다.

### Q: 앵커 가이드를 안 쓰면 어떻게 되나요?

**A**: GPT가 Phase 1 분석 결과만으로 프롬프트를 설계합니다. 앵커 가이드가 있으면 "이 요약 과제에서 반드시 지켜야 할 원칙"을 고정할 수 있어 더 안정적인 개선이 가능합니다.

### Q: 추론 수준(Reasoning)은 어떤 걸 선택해야 하나요?

**A**:
- **첫 Run, 복잡한 분석**: High (정밀) 권장
- **반복 실험, 빠른 검증**: Medium (균형)
- **Phase 4 Judge**: 항상 Low로 고정 (GPT Judge는 빠른 판정이 적합)

### Q: 점수가 안 올라가요

**A**: 다음을 점검하세요:
1. Phase 5에서 **회귀 케이스**를 확인 — 어떤 케이스가 나빠졌는지
2. Phase 6의 **해로운 요소**를 확인 — 어떤 변경이 역효과를 냈는지
3. **앵커 가이드** 추가 검토 — 요약 과제의 핵심 원칙이 고정되어 있는지
4. **전략 가이드** 직접 입력 — "~에 집중하라", "~를 하지 마라" 등 방향 지시
5. **커스텀 후보** 직접 작성 — GPT 생성 후보 대신 수동으로 프롬프트 조정
