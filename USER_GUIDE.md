# Prompt Improver 사용자 가이드

> LLM 요약 프롬프트를 반복 실험으로 개선하는 도구입니다.
> Judge JSON(정답 판정 데이터)을 업로드하면, GPT가 오류를 분석하고 새로운 프롬프트를 설계하며, Dify 워크플로우에서 실행 후 재판정하여 점수를 비교합니다.

## 1. 전체 흐름 요약

```
실험(Task) 생성
 └─ Run 생성 (Zero 또는 Continue)
     ├─ Phase 1: 오류 분석 ─ Judge JSON 업로드 → GPT가 오답 원인 분석
     ├─ Phase 2: 프롬프트 설계 ─ GPT가 개선된 프롬프트 후보 3개 생성 + Mini-validation
     ├─ Phase 3: Dify 실행 ─ 선택한 후보를 Dify 워크플로우에서 전체 케이스 실행
     ├─ Phase 4: Judge 재판정 ─ 새로 생성된 요약을 GPT Judge가 판정
     ├─ Phase 5: 성능 비교 ─ 이전 Run과 점수 비교, 개선/회귀 분석
     └─ Phase 6: 전략 피드백 ─ 무엇이 효과적이었고, 다음에 무엇을 바꿔야 하는지 분석
          └─ 다음 Run 시작 (Continue 모드) → Phase 6 피드백이 자동 주입
```

**목표**: LLM Judge 기준 **정답 + 과답 ≥ 95%** 달성

## 2. 시작하기: 실험 생성

좌측 사이드바에서 **"+ 새 실험 만들기"** 버튼을 클릭합니다.

### 필수 입력
| 필드 | 설명 |
|------|------|
| **Task명** | 실험 이름 (예: "민원내용 요약 개선") |

### 선택 입력
| 필드 | 설명 |
|------|------|
| **설명** | 실험 목적이나 메모 |
| **요약 유형** | 이 실험에서 다루는 요약 종류 (예: "민원내용", "불편사항 요약"). Phase 1~2에서 GPT가 이 정보를 참고하여 분석합니다. |

### 고급 설정 (접이식)

#### ▸ LLM 설정 (분석 모델)
GPT 분석에 사용할 모델을 지정합니다. **비워두면 서버 기본값**(gpt-oss-120B)을 사용합니다.

| 필드 | 설명 |
|------|------|
| API Base URL | OpenAI 호환 API 엔드포인트 |
| API Key | API 인증 키 |
| 모델명 | 모델 ID (예: gpt-oss-120b-26) |

#### ▸ 시뮬레이션 LLM 설정 (생성 모델)
Phase 2 Mini-validation에서 워크플로우 시뮬레이션에 사용할 **생성 모델**을 지정합니다.
실제 Dify 워크플로우에서 사용하는 모델과 **동일한 모델**을 지정해야 정확한 검증이 가능합니다.

| 필드 | 설명 |
|------|------|
| API Base URL | 생성 모델 API 엔드포인트 |
| API Key | API 인증 키 |
| 모델명 | 모델 ID (예: qwen3.5-35B-A3B) |

#### ▸ 앵커 가이드 (Phase 2 전략 고정 가이드)
GPT가 프롬프트를 설계할 때 **항상 참조**하는 고정 가이드입니다.
`prompts/anchors/` 폴더에 `.txt` 파일로 저장된 가이드 중 하나를 선택합니다.

> **사용 예시**: "민원내용" 요약에서 "대화 요약이 아닌 고객 의도 추출"이 핵심이라는 가이드를 고정하면, GPT가 매 Run마다 이 원칙을 반영하여 프롬프트를 설계합니다.

## 3. Run 생성

실험을 선택한 후, 상단 **"+ 새 Run"** 버튼을 클릭합니다.

### 시작 모드

| 모드 | 설명 |
|------|------|
| **처음부터 (Zero)** | 새로운 Judge JSON을 업로드하고 처음부터 분석합니다. |
| **이전 Run 이어서 (Continue)** | 이전 Run의 Phase 4 판정 결과를 자동으로 Judge JSON으로 사용합니다. Phase 6 피드백도 자동 주입됩니다. |

> **권장**: 첫 번째 Run은 Zero, 이후 Run은 Continue로 반복 실험합니다.

## 4. Phase 1: 오류 분석

### 무엇을 하나요?
업로드된 Judge JSON의 **오답/과답 케이스를 GPT가 분석**하여, 왜 틀렸는지 원인을 분류합니다.

### 사용 방법
1. **Judge JSON 업로드**: Judge가 판정한 결과 JSON 파일을 드래그 또는 클릭으로 업로드
2. **현재 프롬프트 업로드 (선택)**: 현재 사용 중인 요약 프롬프트 TXT를 업로드하면, GPT가 "프롬프트에 이런 지시가 누락/위반되었다"까지 분석합니다.
3. **추론 수준 선택 후 "Phase 1 실행"** 클릭

### 추론 수준(Reasoning)
| 수준 | 설명 |
|------|------|
| High (정밀) | GPT가 깊이 사고. 분석이 정확하지만 느림 (권장) |
| Medium (균형) | 속도와 정확도 균형 |
| Low (빠름) | 빠르지만 분석이 얕을 수 있음 |

### 내부 동작
```
1. Judge JSON 파싱 → 정답/오답/과답 분류
2. 정답 케이스: 즉시 결과 저장 (GPT 분석 불필요)
3. 오답/과답 케이스: 최대 5건씩 병렬로 GPT 분석
   → 각 케이스마다: STT, Reference, Generated, Judge판정을 GPT에 전달
   → GPT가 반환: 버킷(원인), 분석사유, 누락지시, 위반지시, 오류패턴, 개선제안
4. 전체 패턴 요약: 모든 분석 결과를 종합하여 GPT가 top_issues, 개선 방향 도출
5. Reference 요약 기준 분석: 상담사(Reference)가 어떤 기준으로 요약하는지 패턴 추출
```

### 화면 설명

#### 점수 카드 (4개)
| 카드 | 의미 |
|------|------|
| **정답+과답%** | 핵심 지표. 이 비율을 95% 이상으로 올리는 것이 목표 |
| **정답%** | Reference와 동일한 핵심 내용을 요약한 비율 |
| **과답%** | 맞지만 불필요한 내용이 추가된 비율 |
| **오답%** | 핵심 내용이 누락되거나 틀린 비율 |

#### 판정 분포 차트
- **왼쪽**: 정답/과답/오답 건수 막대 그래프
- **오른쪽**: 오류 원인 분류 (버킷) 분포

#### 버킷 설명
| 버킷 | 색상 | 의미 |
|------|------|------|
| **STT 오류** | 분홍 | STT(음성인식) 자체의 오전사가 원인. 프롬프트로 해결 불가 |
| **프롬프트 누락** | 노랑 | 프롬프트에 지시가 없어서 발생. **프롬프트 개선으로 해결 가능** |
| **모델 동작** | 파랑 | 프롬프트 지시는 있지만 모델이 따르지 않음 |
| **Judge 이견** | 보라 | 사람이 보면 맞는데 Judge가 오답으로 판정 |

> **핵심**: "프롬프트 누락" 비율이 높을수록 Phase 2에서 개선 여지가 큽니다.

#### 케이스 테이블
각 행을 클릭하면 상세 분석을 볼 수 있습니다.

| 컬럼 | 설명 |
|------|------|
| ID | 케이스 고유 ID |
| Judge판정 | 정답(녹색)/과답(노란색)/오답(빨간색) |
| 버킷 | 오류 원인 분류 |
| 분석사유 | GPT가 분석한 오류 원인 요약 |
| STT불확실표현 | STT에서 의미가 불명확한 부분 |
| STT | 원본 음성인식 텍스트 |
| Reference | 상담사가 작성한 정답 요약 |
| Generated | LLM이 생성한 요약 |
| Judge이견 | Judge 판정에 의문이 있는 경우 이유 |

> **다운로드**: "JSON ⬇" 또는 "XLSX ⬇" 버튼으로 분석 결과를 내려받을 수 있습니다.

## 5. Phase 2: 프롬프트 설계

### 무엇을 하나요?
Phase 1 분석 결과를 바탕으로 GPT가 **개선된 프롬프트 후보 3개**를 자동 설계합니다.
선택적으로 Mini-validation을 통해 후보를 사전 검증합니다.

### 사용 방법
1. (선택) **전략 가이드** 입력: "Chain of thought 사용 금지" 등 설계 방향 지시
2. **"Phase 2 실행"** 클릭
3. 생성된 후보 카드에서 **"후보 선택"** 클릭
4. 또는 **커스텀 후보 작성** 섹션에서 직접 프롬프트를 작성

### 내부 동작 (4단계)
```
Step 1/4: 전략 수립
  GPT가 Phase 1 분석 + 실험 이력 + 앵커 가이드를 종합하여
  3가지 서로 다른 전략 방향(노드 구조, 역할 분담)을 설계

Step 2/4: 후보별 프롬프트 생성 (병렬)
  각 전략에 맞는 구체적인 system_prompt + user_prompt를 GPT가 작성
  대표 오류 케이스를 참조하여 동일 오류가 반복되지 않도록 지시

Step 3/4: 검증 및 보정
  생성된 프롬프트에 누락된 노드가 있으면 GPT로 자동 보정

Step 4/4: Mini-validation (시뮬레이션 LLM이 설정된 경우)
  ├─ Phase 1에서 발견된 오답 중 3건을 선별
  ├─ 각 후보의 워크플로우를 시뮬레이션 모델(Qwen 등)로 직접 실행
  ├─ Judge 프롬프트로 결과를 판정 → pass_rate 계산
  ├─ pass_rate > 0인 후보만 유지
  └─ 모든 후보 실패 시:
       ├─ 실패 피드백을 GPT에 전달하여 프롬프트 재생성 (1회 재시도)
       └─ 재시도에서도 실패 시: 최선 1개 추천
```

### 화면 설명

#### 후보 카드
각 후보(A, B, C)의 카드에는 다음 정보가 표시됩니다:

| 영역 | 설명 |
|------|------|
| **헤더** | 후보 라벨 + 노드 수 + 모드(explore/converge) |
| **설계 근거** | GPT가 이 전략을 선택한 이유 |
| **노드 탭** | 멀티노드인 경우 Node A / Node B / Node C 탭 전환 |
| **System / User 탭** | 각 노드의 시스템 프롬프트와 사용자 프롬프트 |
| **입출력 변수** | 각 노드가 받는 입력과 생성하는 출력 변수 |
| **"후보 선택" 버튼** | 이 후보를 Phase 3에서 사용할 프롬프트로 선택 |
| **"복사" 버튼** | 이 후보의 프롬프트를 커스텀 편집기에 복사 |

#### Mini-Validation 결과 패널
Mini-validation이 실행된 경우 후보별 검증 결과를 테이블로 확인할 수 있습니다.

| 영역 | 설명 |
|------|------|
| **후보별 그룹** | 각 후보(A, B, C)의 pass_rate와 통과 건수 헤더. pass_rate에 따라 초록/노랑/빨강 배경 |
| **케이스 테이블** | 각 검증 케이스의 판정(정답/과답/오답)과 생성된 결과 미리보기 |
| **행 클릭** | 생성 결과 전체(최대 200자)를 펼쳐서 확인 |
| **접기/펼치기** | 패널 헤더 클릭으로 토글 (기본 펼침) |

> **실행 로그**: SSE 로그에도 케이스별 상세 판정이 출력됩니다 (예: `└ 케이스 case_1: 정답 — "통화 내용 요약에 따르면..."`)

#### 커스텀 후보 작성
GPT 생성 후보를 수정하거나 완전히 새로운 프롬프트를 작성할 수 있습니다.

| 설정 | 설명 |
|------|------|
| **노드 수** | 1~3개. 단일 노드로 충분하면 1개로 |
| **Reasoning** | 체크하면 해당 노드에서 GPT가 추론(thinking) 후 응답 |
| **System Prompt** | 역할 정의, 규칙, 제약조건 |
| **User Prompt** | 실제 데이터 처리 템플릿. `{stt}` 등 변수 사용 |
| **입력 변수** | 이 노드가 필요로 하는 변수 (첫 노드는 반드시 `stt` 포함) |
| **출력 변수** | 이 노드가 생성하는 변수 (마지막 노드는 반드시 `generated`) |

### Design Mode

| 모드 | 조건 | 설명 |
|------|------|------|
| **Explore** | 첫 번째 Run | 완전히 다른 3가지 방향으로 탐색 |
| **Converge** | 2번째 Run~ | 이전 최고 성적 프롬프트를 기반으로 점진 개선 |

### Learning Rate
이전 실험 성적에 따라 변경 폭을 자동 조절합니다:

| 값 | 의미 |
|------|------|
| explore | 처음이므로 자유 탐색 |
| major | 성적이 낮거나 정체 → 큰 구조 변경 |
| medium | 중간 성적 → 요소 재조합 |
| minor | 높은 성적 → 미세 조정만 |

## 6. Phase 3: Dify 실행

### 무엇을 하나요?
Phase 2에서 선택한 프롬프트를 **Dify 워크플로우에 배포**하고, 전체 케이스를 실행하여 새로운 요약을 생성합니다.

### 사용 방법
1. **선택된 후보 확인**: 상단에 선택된 후보의 구조와 프롬프트가 표시됩니다
2. **Dify Workflow Object ID 입력**: Dify에서 생성한 워크플로우의 ID를 입력
3. **"연결 확인"** 클릭: 워크플로우가 정상 동작하는지 검증
4. 연결 확인 후 **"전체 케이스 실행"** 클릭

### 내부 동작
```
1. Phase 1의 전체 케이스에서 STT를 Dify 워크플로우에 입력
2. 최대 5건씩 병렬 실행 (각 케이스 타임아웃: 120초, 재시도: 3회)
3. Dify 응답에서 추출:
   - generated: 최종 요약 텍스트
   - 중간 출력: 각 노드가 생성한 중간 결과
4. 결과를 DB에 저장
```

### 화면 설명

| 영역 | 설명 |
|------|------|
| **선택된 후보** | 후보 라벨, 노드 수, 설계 근거, 각 노드 프롬프트 |
| **노드 상세** | 클릭하면 각 노드의 입력변수, System/User 프롬프트, 출력변수 확인 |
| **Dify 연결** | Object ID 입력 + 연결 확인 상태 (✓ 녹색/✕ 빨간색) |
| **진행 바** | 전체 케이스 실행 진행률 |

> **주의**: Dify에 워크플로우를 먼저 구성해야 합니다. Phase 2에서 설계된 노드 구조(시스템/유저 프롬프트, 입출력 변수)를 Dify 워크플로우에 동일하게 설정하세요.

---

## 7. Phase 4: Judge 재판정

### 무엇을 하나요?
Phase 3에서 새로 생성된 요약을 **GPT Judge가 다시 판정**합니다.
이전 Run과 동일한 Judge 기준으로 평가하여, 프롬프트 변경 효과를 객관적으로 측정합니다.

### 사용 방법
**"Phase 4 실행"** 클릭만 하면 됩니다.

### 내부 동작
```
1. Phase 3에서 generated가 생성된 케이스만 로드
2. 각 케이스마다 Judge 프롬프트로 GPT 호출 (병렬, 최대 5건)
   입력: STT + Reference + Keywords + Generated
   출력: 판정(정답/과답/오답) + 사유
3. JSON 파싱 실패 시 텍스트 분류로 폴백:
   - "결론 상반" 감지 → 오답
   - 키워드 매칭 (한국어→영어 순서)
4. 점수 집계: 정답%, 과답%, 오답%, 정답+과답%
5. 이전 Run과 Delta 계산 (개선/회귀/변화없음)
```

### 화면 설명

| 영역 | 설명 |
|------|------|
| **점수 카드** | Phase 1과 동일한 4개 카드. 새로운 프롬프트의 성적 |
| **케이스 테이블** | 각 케이스의 판정, 사유, STT, Reference, Generated |

> **행 클릭 시**: 중간 출력(각 노드가 생성한 중간 결과)까지 확인 가능

---

## 8. Phase 5: 성능 비교

### 무엇을 하나요?
Phase 4 결과를 **이전 Run과 비교**하여, 프롬프트 변경이 실제로 효과가 있었는지 분석합니다.

### 화면 설명 (실행 버튼 없음 — Phase 4 완료 시 자동 계산)

#### 점수 카드
이번 Run의 점수 (Phase 4와 동일)

#### 목표 달성 배너
- **녹색**: "목표 달성!" (정답+과답 ≥ 95%)
- **노란색**: "목표까지 X%p 부족"

#### Run별 성능 추이 차트
모든 Run의 정답+과답% 변화를 라인 차트로 보여줍니다.
> 첫 번째 Run에서는 "다음 Run부터 추이 비교가 가능합니다" 메시지가 표시됩니다.

#### Delta 카드 (3개)
| 카드 | 의미 |
|------|------|
| **개선** (녹색) | 이전에 오답이었는데 이번에 정답/과답으로 바뀐 케이스 수 |
| **회귀** (빨간색) | 이전에 정답/과답이었는데 이번에 오답으로 바뀐 케이스 수 |
| **변화없음** (회색) | 판정이 동일한 케이스 수 |

> **핵심**: 개선 > 회귀여야 프롬프트 변경이 효과적입니다.

#### 회귀 케이스 테이블
회귀된 케이스만 모아서 보여줍니다. 이전 판정 → 현재 판정과 Judge 사유를 확인하세요.

#### 전체 케이스 테이블
모든 케이스의 판정 + 변화(개선/회귀/변화없음/신규) 정보를 포함합니다.

| 변화 배지 | 의미 |
|------|------|
| **개선** (녹색) | 오답 → 정답/과답 |
| **회귀** (빨간색) | 정답/과답 → 오답 |
| **변화없음** (회색) | 동일 |
| **신규** (파란색) | 이전 Run에 없던 케이스 |

---

## 9. Phase 6: 전략 피드백

### 무엇을 하나요?
이번 Run의 결과를 종합 분석하여, **다음 Run에서 무엇을 바꿔야 하는지** GPT가 피드백을 생성합니다.
이 피드백은 다음 Run (Continue 모드)의 Phase 2에 자동으로 주입됩니다.

### 사용 방법
**"Phase 6 실행"** 클릭

### 내부 동작
```
1. 현재 프롬프트 vs 이전 프롬프트 구조 비교 (노드 수, reasoning, 프롬프트 변경점)
2. Learning rate 계산 (점수 추이 기반)
3. Delta 분석 (개선/회귀 케이스의 중간 출력까지 비교)
4. GPT에게 다음 정보 전달:
   - 프롬프트 변경 전후 비교
   - 개선/회귀 케이스 상세
   - 중간 노드 출력 분석
   - Reference 요약 기준
   - 전체 실험 이력
5. GPT가 반환:
   - Backprop 분석: 어떤 변경이 어떤 케이스에 영향을 미쳤는지
   - 효과적 요소: 유지해야 할 변경점
   - 해로운 요소: 제거해야 할 변경점
   - 다음 방향: 구체적인 개선 방향
   - 제약사항: 절대 바꾸지 말아야 할 요소
```

### 화면 설명

| 영역 | 설명 |
|------|------|
| **Learning Rate** | 파란 배지. 다음 Run에서의 변경 폭을 나타냄 |
| **Backprop 분석** | 프롬프트 변경 → 케이스별 영향 분석 텍스트 |
| **효과적 요소** (녹색 태그) | 다음 Run에서 유지해야 할 변경점 |
| **해로운 요소** (빨간 태그) | 다음 Run에서 제거해야 할 변경점 |
| **다음 방향** | 다음 Run의 구체적 개선 방향 |

> Phase 6 완료 후 **"다음 Run 시작 →"** 버튼이 나타납니다.

---

## 10. 다음 Run 시작 (반복 개선)

Phase 6 완료 후 **"다음 Run 시작 →"** 클릭 시:

1. **Continue 모드**로 새 Run 생성
2. Phase 4 판정 결과가 자동으로 Judge JSON이 됨
3. Phase 6 피드백이 Phase 2에 자동 주입
4. 누적된 효과/해로운 패턴, 제약사항이 모두 반영

> **반복할수록**: GPT가 이전 실험에서 학습한 내용을 누적 반영하므로, 점수가 점진적으로 개선됩니다.

## 11. 용어 정리

| 용어 | 설명 |
|------|------|
| **Task (실험)** | 하나의 요약 유형에 대한 최상위 실험 단위 |
| **Run** | Task 내에서의 한 번의 실험 사이클 (Phase 1~6) |
| **Judge JSON** | LLM Judge가 판정한 결과 파일. 각 케이스의 정답/오답/과답 판정 포함 |
| **STT** | Speech-to-Text. 음성인식 원본 텍스트 |
| **Reference** | 상담사가 작성한 정답 요약 |
| **Generated** | LLM이 생성한 요약 |
| **정답** | Generated가 Reference의 핵심 내용을 정확히 담고 있음 |
| **과답** | 맞긴 하지만 불필요한 내용이 추가됨 |
| **오답** | 핵심 내용이 누락되거나 잘못됨 |
| **버킷** | 오류 원인 분류 (STT오류, 프롬프트누락, 모델동작, Judge이견) |
| **Learning Rate** | 다음 Run에서 프롬프트를 얼마나 크게 바꿀지 (explore/major/medium/minor) |
| **Delta** | 이전 Run과 현재 Run의 케이스별 판정 변화 (개선/회귀/변화없음) |
| **앵커 가이드** | Phase 2에서 GPT가 항상 참조하는 고정 가이드 |
| **Mini-validation** | Phase 2에서 후보 프롬프트를 소수 케이스로 사전 검증하는 단계 |
| **Dify** | LLM 워크플로우 실행 플랫폼 |
| **Reasoning** | GPT의 추론(thinking) 기능. 켜면 더 정확하지만 느림 |
