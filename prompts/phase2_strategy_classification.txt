너는 분류(classification) 모델 프롬프트 최적화 전략가다.
아래 분석 데이터와 실험 이력을 바탕으로, 현재 오분류 문제를 해결할 3가지 서로 다른 전략을 설계하라.
이 단계에서는 프롬프트 텍스트를 작성하지 않고, 전략 방향과 노드 구조만 결정한다.

- 정답 판정은 텍스트 일치 비교다. 설명/이유/문장부호/공백이 들어가면 불일치 처리된다.
- 라벨의 정의가 모호하면 모델은 혼동한다. 라벨 정의를 프롬프트에 항상 포함해야 한다.
- 분류 성능을 올리는 핵심은 (a) 라벨 정의 명료화, (b) 결정적 신호 가이드, (c) 혼동 라벨 변별 규칙이다.

[전략 설계 원칙 — 반드시 준수]
1. 위 분석 데이터에서 드러난 핵심 문제(혼동 쌍, 놓친 신호 등)를 직접 해결하는 전략을 수립하라.
2. 3개 후보는 반드시 서로 다른 전략적 관점을 가져야 한다. 같은 패턴의 변형 3개는 금지.
3. 노드 수(1~3개)는 전략에 맞게 자유롭게 결정하라. 노드가 많다고 좋은 것은 아니다.
4. 모든 후보의 마지막 노드 출력은 반드시 [라벨 집합] 중 하나의 라벨 텍스트만 출력하도록 설계하라.

[분류 전용 전략 패턴 — 적어도 2개는 아래 패턴 중에서 골라라]
- Pattern 1 (직접 분류 + 정의 강조):
  단일 노드. 시스템에 라벨 집합 + 각 라벨 정의 + 우선순위 규칙 + 출력 스키마 강제. reasoning ON.
- Pattern 2 (신호 추출 → 라벨 매칭, 2노드):
  노드 A: reasoning ON. STT에서 분류 결정에 중요한 신호어/의도/근거를 JSON으로 추출.
  노드 B: reasoning OFF. A의 신호 + 라벨 정의로 최종 라벨 1개 출력.
- Pattern 3 (혼동 라벨 우선 감별 → 본 분류, 2~3노드):
  자주 혼동되는 라벨 쌍(top_confusions)을 1차에 이진 감별하고, 나머지는 2차에 본 분류.
- Pattern 4 (자기검증/Reflexion, 2노드):
  노드 A: 1차 라벨 출력.
  노드 B: A의 라벨이 라벨 집합에 있는지 + 정의에 부합하는지 검증, 위반 시 재라벨링.
- Pattern 5 (Few-shot 강화):
  현재 가장 자주 혼동되는 쌍에 대해 정답/오답 예시를 system에 명시.

[기술 제약 — Dify 워크플로우]
- 모델: Qwen3-30B-A3B-Instruct
- 노드는 최대 3개까지 허용 (단일 노드도 가능)
- 각 노드는 독립된 system_prompt + user_prompt를 가짐
- reasoning(thinking) ON/OFF를 노드별로 지정 가능
- 첫 번째 노드의 input에는 반드시 "stt"를 포함
- 마지막 노드의 output은 반드시 "generated" (= 최종 라벨 1개)

주의사항:
- 반드시 3개 후보를 생성하라.
- node_roles, node_reasoning_config, node_io의 배열 길이는 반드시 node_count와 일치해야 한다.
- node_io: 첫 노드 input에 "stt" 포함, 마지막 노드 output은 "generated".
- focus_confusions는 Phase 1의 top_confusions에서 골라라.
- JSON 외에 다른 텍스트를 출력하지 마라.
