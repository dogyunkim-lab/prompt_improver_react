[전략 설계 원칙 — 반드시 준수]
1. 위 분석 데이터에서 드러난 핵심 문제를 직접 해결하는 전략을 수립하라.
2. 특정 워크플로우 패턴에 편향되지 마라. 다양한 접근이 가능하다:
   - 직접 지시 강화: 명확한 규칙·기준·예시로 LLM의 판단을 가이드
   - 단계적 추론: reasoning을 활용해 판단 근거를 먼저 생성한 뒤 최종 출력
   - 분류→맞춤 처리: 케이스 유형을 먼저 분류하고 유형별 전략을 적용
   - 초안→자기검증: 초안 생성 후 기준 대비 누락/초과를 스스로 검증·수정
   - 구조화 출력: 정형 템플릿으로 필수 항목 누락을 구조적으로 방지
   - 또는 분석 데이터가 시사하는 완전히 새로운 접근
3. 3개 후보는 반드시 서로 다른 전략적 관점을 가져야 한다.
   같은 패턴(예: "추출→나열")의 변형을 3개 만드는 것은 금지한다.
4. 노드 수(1~3개)는 전략에 맞게 자유롭게 결정하라. 노드가 많다고 좋은 것이 아니다.
   단일 노드로 충분한 전략이면 1개로 설계하라.

[기술 제약 — Dify 워크플로우]
- 모델: Qwen3-30B-A3B-Instruct
- 노드는 최대 3개까지 허용 (단일 노드도 가능)
- 각 노드는 독립된 system_prompt + user_prompt를 가짐
- reasoning(thinking) ON/OFF를 노드별로 지정 가능
- 첫 번째 노드의 input에는 반드시 "stt"를 포함
- 마지막 노드의 output은 반드시 "generated"

[출력 형식 - JSON만 출력]
{{
  "candidates": [
    {{
      "label": "A",
      "strategy_name": "이 전략의 핵심 아이디어를 한 문장으로 요약",
      "node_count": 1,
      "node_roles": ["이 노드가 하는 일을 한 문장으로"],
      "node_reasoning_config": [true],
      "node_io": [
        {{"input": ["stt"], "output": "generated"}}
      ],
      "rationale": "분석 데이터의 어떤 문제를 어떻게 해결하는지 (2~3문장)",
      "focus_patterns": ["이 후보가 해결하려는 error_pattern 키워드"]
    }}
  ],
  "design_summary": "전체 설계 전략 요약 (learning rate 반영)"
}}

주의사항:
- 반드시 3개 후보를 생성하라.
- node_roles, node_reasoning_config, node_io의 배열 길이는 반드시 node_count와 일치해야 한다.
- node_io: 각 노드의 입력 변수(input)와 출력 변수(output)를 명시하라.
  첫 번째 노드의 input에는 반드시 "stt"를 포함하고, 마지막 노드의 output은 반드시 "generated"여야 한다.
- focus_patterns는 Phase 1의 error_pattern_ranking에 있는 패턴 키워드를 참조하라.
- JSON 외에 다른 텍스트를 출력하지 마라.
