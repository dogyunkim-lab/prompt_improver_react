[프롬프트 작성 원칙 — 분류 전용]
1. 마지막 노드의 system_prompt에는 반드시 다음을 포함하라:
   - [라벨 집합] 전체 텍스트 (정확히 위 그대로)
   - 각 라벨의 정의 (반드시 위 정의를 그대로 또는 명료하게 다듬어 포함)
   - "출력은 반드시 위 라벨 중 정확히 하나만, 다른 어떤 문자도 추가하지 말고 출력하라" 라는 명시적 출력 제약
2. 자주 혼동되는 라벨 쌍(focus_confusions)에 대해서는 두 라벨의 변별 규칙을 명시하라.
   예) "환불/취소 요구가 명시되면 '민원', 단순 정보 요청이면 '문의'"
3. 대표 오분류 케이스에서 모델이 놓친 신호와 과대해석한 신호를 반영하여 가이드를 작성하라.
4. 멀티 노드 전략이라면 중간 노드는 신호 추출/감별/검증 등 역할을 분담하고,
   마지막 노드만이 [라벨 집합]에서 1개를 골라 출력하도록 설계하라.
5. system_prompt에는 역할/규칙/라벨 집합/정의/변별 규칙/출력 제약을 넣어라.
6. user_prompt에는 STT(또는 직전 노드의 출력)를 받아 처리하는 템플릿을 넣어라.

[STT 오전사 헷지 규칙 — 반드시 포함]
모든 노드 프롬프트에 아래 지시를 포함하라:
"STT에서 문맥상 의미가 불명확한 표현은 임의로 해석하지 말고, 분명한 신호어 위주로 판단하라."

[출력 형식 - JSON만 출력]
{{
  "candidate_label": "{candidate_label}",
  "nodes": [
    {{
      "node_label": "A",
      "system_prompt": "이 노드의 시스템 프롬프트 (역할 정의, 라벨 집합, 라벨 정의, 변별 규칙, 출력 제약)",
      "user_prompt": "이 노드의 사용자 프롬프트 템플릿 ({{{{stt}}}} 등 변수 사용)",
      "input_vars": ["stt"],
      "output_var": "전략에 맞는 서술적 변수명 또는 generated",
      "reasoning": true
    }}
  ]
}}

주의사항:
- nodes 배열의 길이는 반드시 {node_count}개여야 한다.
- node_label은 순서대로 A, B, C를 사용하라.
- system_prompt: 노드 역할/규칙/라벨 집합/정의/변별 규칙/출력 제약. Dify의 SYSTEM 필드.
- user_prompt: 입력 데이터 처리 템플릿. Dify의 USER 필드. 변수는 {{{{변수명}}}} 형태.
- input_vars: 첫 노드는 반드시 "stt" 포함. 후속 노드는 이전 노드의 output_var를 포함 가능.
- output_var: 마지막 노드는 반드시 "generated". 중간 노드는 서술적 이름(예: signals, candidate, verified).
- 마지막 노드의 system_prompt에는 반드시 [라벨 집합]의 모든 라벨 텍스트를 정확히 그대로 포함하라.
- JSON 외에 다른 텍스트를 출력하지 마라.
