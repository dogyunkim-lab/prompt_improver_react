[요약 앵커 가이드 — "불편사항" 요약 Task]

1. 요약 대상의 본질
   "불편사항"은 고객이 이 통화에서 표현한 서비스 이용 불편·불만을 압축한 것이다.
   만족도 조사 후처리 양식의 "불편사항" 란에 기입할 내용이라고 생각하라.
   고객이 무엇을 불편하게 느꼈는지, 그 현상과 감정을 사실대로 기록한다.

2. 요약 구조
   "[불편 현상/원인] + [고객 감정 표현]" 형태로 작성한다.
   - Reference 예시:
     "청구시 사진첨부를 할 때 한번에 첨부가 안되고 한장씩 첨부해야해서 불편했고 로그인도 번거로움"
     "핸드폰으로 하는거 너무 짜증났다함"
     "1달만에 보험금 지급되어 불만"
   - 불편 현상과 감정 표현이 모두 포함되어야 한다.
   - 다수의 불편 항목이 있으면 "/" 또는 줄바꿈으로 나열하되, Reference에 있는 항목만 포함한다.

3. 추출 우선순위 — STT에서 무엇을 뽑을 것인가
   ① 불편의 핵심 현상 (가장 중요)
     → "한번에 첨부 안됨", "다음으로 안넘어감", "인증완료 안됨", "전화연결 안됨"
     → 구체적 현상을 그대로 기술할 것. 일반화하면 오답.
   ② 고객의 감정 표현 (반드시 포함)
     → "불편", "불만", "짜증", "화남", "억울", "기분 안좋음"
     → 이 감정 키워드를 생략하면 오답이다. 절대 빠뜨리지 말 것.
   ③ 불편이 발생한 채널/시스템
     → "앱", "AIA+", "콜센터", "ARS", "카카오톡", "챗봇"
     → 채널명이 Reference 키워드에 포함되면 정확히 기재.
   ④ 구체적 수치/조건
     → "1달만에", "두번세번", "1천만원", "한장씩"
   ⑤ 이하 생략: 상담사 안내/대응, 본인확인 절차, 사건 경위 상세, 다른 보험사 비교 설명

4. 금지 사항
   — 키워드/감정 관련 —
   - 감정 키워드를 격식체로 바꾸지 말 것
     ("짜증" → "불편함" ✗, "불만" → "아쉬움" ✗, "화남" → "우려" ✗)
   - 불편 현상을 추상화하지 말 것
     ("한번에 첨부 안됨" → "사진 전송 과정의 어려움" ✗)
   - 채널/시스템명 생략 금지
     ("앱 본인확인 단계에서..." → "본인 확인 단계에서..." ✗)

   — 과답 방지 —
   - Reference에 없는 추가 불편사항을 STT에서 가져오지 말 것
     (과답의 가장 흔한 원인: STT에 여러 불편이 언급되지만 Reference는 특정 항목만 선별)
   - 상담사의 대응·안내·약속을 불편사항에 포함하지 말 것
   - 대화 배경이나 사건 경위를 서술하지 말 것

   — 문체 관련 —
   - 고객의 구어체 표현을 살리되, 간결하게 압축할 것
   - Reference처럼 "~불편", "~불만", "~짜증남" 등 감정으로 마무리

5. STT 오전사 대응
   STT는 음성 인식 결과이므로 오탈자, 문맥 단절이 있을 수 있다.
   - 문맥상 유추 가능하면 원래 의도에 맞게 자연스럽게 요약
   - 의미를 알 수 없는 부분은 생략
   - 보험/시스템 용어는 정확한 표현으로 교정하되 의미를 바꾸지 말 것

6. 자주 틀리는 패턴과 교정

   [감정 키워드 누락 — 오답 최다 원인]
   ✗ "핸드폰으로 진행하는 서비스 이용 시 시간이 길어짐"
   ✓ "핸드폰으로 하는거 너무 짜증났다함"

   ✗ "보험금 청구 후 처리 기간이 1개월 소요됨"
   ✓ "1달만에 보험금 지급되어 불만"

   [채널/시스템 생략]
   ✗ "본인확인 단계에서 다음으로 진행되지 않음"
   ✓ "앱 본인확인 단계에서 다음으로 안넘어가고 오류내용없이 고객센터 전화하라고 나와 불편"

   [불필요 정보 추가 — 과답]
   ✗ "보험금 청구 시 서류 업로드가 복잡함 + 고객센터 전화 연결이 어려움 + 통화 대기 시간이 김"
   ✓ "컴퓨터, 핸드폰을 잘 못하니 서류 업로드나 글씨 읽으며 사진 넣는게 복잡함"
     (Reference에 없는 "전화 연결 어려움" 제거)
