[프롬프트 작성 원칙]
1. 위 전략 방향을 충실히 따르라. 전략의 핵심 아이디어를 프롬프트에 직접 반영하라.
2. 대표 오류 케이스를 분석하여, 해당 오류가 재발하지 않도록 구체적 지시를 포함하라.
3. 특정 패턴(예: 키워드 추출→나열)에 편향되지 말고, 전략이 의도하는 방식대로 작성하라.
4. system_prompt에는 역할 정의, 판단 기준, 규칙, 제약조건을 넣어라.
5. user_prompt에는 실제 입력 데이터를 처리하는 템플릿을 넣어라.

[STT 오전사 헷지 규칙 - 반드시 포함]
모든 노드 프롬프트에 아래 지시를 포함하라:
"STT에서 문맥상 의미가 불명확한 표현은 임의로 해석하거나 추론하지 말고
 원문 그대로 유지하거나 해당 부분을 생략하라."

[출력 형식 - JSON만 출력]
{{
  "candidate_label": "{candidate_label}",
  "nodes": [
    {{
      "node_label": "A",
      "system_prompt": "이 노드의 시스템 프롬프트 (역할 정의, 규칙, 제약조건)",
      "user_prompt": "이 노드의 사용자 프롬프트 템플릿 ({{{{stt}}}} 등 변수 사용)",
      "input_vars": ["stt"],
      "output_var": "전략에 맞는 서술적 변수명 또는 generated",
      "reasoning": true
    }}
  ]
}}

주의사항:
- nodes 배열의 길이는 반드시 {node_count}개여야 한다.
- node_label은 순서대로 A, B, C를 사용하라.
- system_prompt: 노드의 역할, 규칙, 제약조건을 정의한다. Dify의 SYSTEM 필드에 넣을 내용이다.
- user_prompt: 실제 입력 데이터를 처리하는 템플릿이다. Dify의 USER 필드에 넣을 내용이다.
  변수는 {{{{변수명}}}} 형태로 표기하라.
- input_vars: 이 노드가 필요로 하는 입력 변수 목록이다. 첫 번째 노드는 반드시 "stt"를 포함해야 한다.
  후속 노드는 이전 노드의 output_var를 input_vars에 포함할 수 있다.
- output_var: 이 노드가 출력하는 변수명이다. 마지막 노드의 output_var는 반드시 "generated"여야 한다.
  중간 노드의 output_var는 역할을 나타내는 서술적 이름을 사용하라 (예: analysis, draft, verified 등).
  사용자는 Dify 워크플로우의 "출력 노드(End)"에 각 노드의 output_var와 동일한 변수명을 등록해야 한다.
  이를 통해 시스템이 각 변수가 어떤 노드에서 생성되었는지 자동으로 추적한다.
- 기존 prompt 필드 대신 반드시 system_prompt + user_prompt 쌍으로 작성하라.
- reasoning 값은 전략의 node_reasoning_config을 따르라.
- JSON 외에 다른 텍스트를 출력하지 마라.
