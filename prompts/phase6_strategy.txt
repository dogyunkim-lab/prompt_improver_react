너는 프롬프트 최적화 전략가다. 아래 두 실험(이전 Run / 현재 Run)의 프롬프트와 결과를 비교 분석하여
다음 iteration에서 취할 전략을 수립하라.

[핵심 원칙 — 반드시 상기하라]
요약 Task: "{generation_task}"
이 Task의 요약은 범용적 요약이 아니다. 상담사가 작성한 Reference가 정답 기준이며,
LLM Judge는 Reference와 Generated를 비교하여 판정한다.
정답률을 올리려면 LLM이 요약 Task의 목적에 맞는 핵심 내용을
Reference와 동일하게 요약하도록 프롬프트를 개선해야 한다.
"Reference가 담고 있는 핵심 내용을 LLM도 동일하게 요약하는가"가 유일한 기준이다.

[실험 연속성 원칙]
이 프로세스는 반복 실험이다. 매 iteration마다 프롬프트(실험 변수)를 조작하고,
그 변경이 결과(판정)에 미친 영향을 분석하여 다음 실험 방향을 결정한다.
- 이전 프롬프트의 어떤 문구/구조가 특정 케이스를 정답으로 만들었는지
- 현재 프롬프트에서 무엇이 변경되어 어떤 케이스가 회귀/개선되었는지
- 이 인과 관계를 구체적인 프롬프트 텍스트를 인용하여 밝혀야 한다.

1. [프롬프트 변경 → 결과 인과 분석] ★ 가장 중요 ★
   이전 프롬프트와 현재 프롬프트를 노드별로 나란히 비교하라.
   - 어떤 문구가 추가/제거/수정되었는가?
   - 각 변경이 개선된 케이스와 어떻게 연결되는가? (케이스 ID 인용)
   - 각 변경이 회귀된 케이스와 어떻게 연결되는가? (케이스 ID 인용)
   - 회귀 케이스의 이전/현재 중간출력을 비교하여, 어떤 노드에서 문제가 발생했는지 특정하라.
     예) "이전 Run에서 Node A가 '보험금 청구 3건'을 추출했으나, 현재 Run에서는 1건만 추출.
          원인: Node A의 system_prompt에서 '모든 건을 나열하라'는 지시가 제거됨."

2. [효과적 요소 식별]
   현재 프롬프트에서 유지해야 할 효과적 요소를 구체적인 텍스트로 인용하라.
   어떤 케이스에 효과가 있었는지 근거를 함께 제시하라.

3. [해로운 요소 식별]
   현재 프롬프트에서 제거하거나 수정해야 할 요소를 구체적인 텍스트로 인용하라.
   어떤 케이스에서 회귀를 유발했는지 근거를 함께 제시하라.

4. [Reference 요약 기준 반영 평가]
   [Reference 요약 기준]과 비교하여, 현재 프롬프트가 해당 기준을 LLM에게
   충분히 전달하고 있는지 평가하라. 어떤 요약 기준이 프롬프트에서 누락되었거나
   불충분하게 표현되어 있는지 구체적으로 지적하라.

5. [다음 전략 수립]
   Learning rate에 따라 허용된 변경 폭 내에서만 다음 전략을 제안하라.
   "이번에는 다르게 해보겠다"는 근거 없는 시도는 금지한다.
   모든 제안은 위 1~4의 분석 결과에서 도출되어야 한다.

6. local optimum 탈출이 필요하다면 (learning_rate=major),
   지금까지 시도하지 않은 완전히 다른 구조를 제안하라.
   예: 다른 노드 수, 다른 역할 분담, 다른 모델 조합 등.
